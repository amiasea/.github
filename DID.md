Wallet: A secure software or hardware container (e.g., a mobile app, HSM, or cloud vault) used by the Holder to store and manage private keys and Verifiable Credentials.
Holder: The "Actor" (Human, AI, or Service) that possesses and presents credentials. The Holder uses their Private Key to generate a Verifiable Presentation, proving they are the rightful owner of the claims.
Issuer: The Authority who signs the data. The Issuer verifies the Actor’s attributes and issues a Verifiable Credential cryptographically bound to the Actor’s Public Key.
Verifier: The "Bouncer" or Resource Provider (e.g., your API or Azure UAMI) that requests a Verifiable Presentation and validates it against a Presentation Definition (Policy) to grant access.
#1: DID Resolution
Input: The Verifier takes the Issuer’s DID (e.g., did:web:university.edu) from the credential.
Lookup: The DID Resolver looks up that ID on its specific registry (for did:web, it is a standard HTTPS fetch of a did.json file).
Output: The resolver returns a DID Document containing the Issuer’s public identity metadata.
Extraction: The Verifier identifies the specific Verification Method (the public key material) within the DID Document to be used for signature validation.
#2: Credential Verification
Signature Validation: The high-level process of using the Issuer's Public Key to confirm the mathematical validity of the digital signature.
Proof Validation: The Verifier checks the proof object attached to the credential. This is where the mathematical "Integrity Check" happens to ensure the signature was generated by the Issuer’s private key.
Data Integrity: The process of re-hashing the Claims (the attributes) and comparing them to the hash inside the validated proof to ensure the data has not been tampered with.
#3: Credential Status Check
Mechanism: The Verifier identifies the Status Method (e.g., a BitstringStatusList or RevocationList2020) referenced in the credential’s credentialStatus property.
Retrieval: The Verifier fetches the Status List (a small, signed, and highly cacheable file) from the URL provided by the Issuer.
Resolution: The Verifier checks the specific Index of the Actor's credential within that list to confirm it is still Active and has not been Revoked or Suspended.
#4: Policy Evaluation
Input: The Verifier takes the Verifiable Presentation (the bundle of claims) provided by the Holder.
Definition: The Verifier compares the claims against its internal Presentation Definition (the policy), which acts as a "filter" for the request. 
Logic Matching: The Verifier evaluates the claims using Input Descriptors (required fields) and Submission Requirements (logical operators like AND/OR).
Business Rules: Beyond simple math, the Verifier applies specific Business Rules—such as checking if a "Clearance Level" claim is strictly greater than 5 or if the "Department" matches the resource's owner.
Decision: If all criteria in the Presentation Definition are met, the Verifier issues an Authorization Token or grants direct access to the protected resource.
